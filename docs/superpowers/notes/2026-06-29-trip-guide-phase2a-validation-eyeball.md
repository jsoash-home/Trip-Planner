## What needs work (the eyeball half)

These are issues only a human eye catches — every one of these would
have passed the script-grep validation, because the markup is correct;
the *rendered* result is the problem.

### Finding 1 — Practical-link affordance is invisible against the dark bg

This is the headline finding and the most concrete Phase 2b input.

The CSS is the spec's:

```css
a.practical-link {
  color: var(--ink);
  text-decoration-color: var(--hairline);
  text-decoration-thickness: 1px;
  text-underline-offset: 2px;
}
```

Resolved colors:

| Token | Hex | RGB | Role |
|---|---|---|---|
| `--bg` | `#0e131a` | (14, 19, 26) | Page background |
| `--hairline` | `#2a323f` | (42, 50, 63) | Underline color |
| `--ink` | `#e9e6dd` | (233, 230, 221) | Link text (same as body ink) |

The underline color differs from the background by ~25–35 per channel.
At normal reading distance against this background it's
indistinguishable from the bg — the underline simply isn't visible.
Combined with the link text rendering in the same color as body ink,
the visual signal that something is clickable is zero until the user
hovers.

This affects every linked surface uniformly:

- **`day_by_day` site cards** — "Sørenga Sjøbad", "Oslo Opera House",
  "Mathallen Oslo" all read as plain `<h5>` headings.
- **`things_to_do`** — "Vigeland Sculpture Park", "Tim Wendelboe",
  "Mathallen Oslo" same. The neighborhood mono tag below the title
  ("Frogner", "Vulkan") is *more visually prominent* than the link
  above it because the accent color carries.
- **`food` where-to-eat** — "Maaemo", every restaurant title.
- **`field_guide`** — landmark/museum entries (wildlife entries
  correctly skip per spec).
- **Bibliography** — "The Northmen's Fury", every source title.
- **Go-deeper cards** — every card title.

The hover state IS well-tuned: thickness jumps to 2px and both color
and decoration-color shift to `--accent` (#7ec8b1). The problem is the
reader has to know to hover *first*, and on a souvenir-grade guide
they may not — the editorial register suggests "essay to read," not
"clickable web page."

**Why the spec arrived here.** The skill explicitly notes
*"Subtle underline + accent on hover already says 'link'; the glyph
adds visual noise without information."* That's defensible against
the noise risk, but the implementation overshot — `--hairline` was
chosen for divider rules and faint borders, where the goal IS for
the line to recede. Underlining with the same near-invisible color
inherits the wrong affordance.

**Phase 2b proposal:**

- Change underline color from `--hairline` to `--ink-soft`
  (`#a8a39a` rgb 168,163,154). That's still recessive against
  body prose (link uses full `--ink` for text + soft for underline)
  but visible — the contrast vs bg is ~150 per channel instead of
  ~30.
- Optionally: bump default `text-decoration-thickness` to `1.5px`
  on dark themes to keep the line legible at small sizes.
- The hover state stays as-is — it's already strong.

This is a CSS-only change in the composer's `<style>` block; no
markup change needed and no impact on light-theme palettes (where
`--hairline` would actually have produced an acceptable underline,
which is presumably why it slipped through).

### Finding 2 — Deep mode currently shows the same content as Standard

Mode-toggle counts at standard vs deep:

```
.deep:               30 / 30 visible in both
.dig-deeper:         0 / 0
.sidenote-content:   0 / 0
.endnotes:           0 / 0
```

The compose script (`scripts/2026-06-27_compose_trip2.py`) doesn't
emit any of the three Deep-only apparatus elements
(`.dig-deeper`, `.sidenote-content`, `.endnotes`). So clicking Deep
literally changes nothing on screen vs Standard.

This isn't a bug in the toggle JS — it's correct. The gap is on the
composer side: the SKILL.md mode-toggle CSS lists four classes that
Deep is supposed to reveal beyond Standard, but the only one the
composer actually emits is `.deep` (which is part of the Standard
view, not the Deep-extra view).

**Phase 2b proposal:**

Either close the gap by emitting `.sidenote-content` and `.dig-deeper`
in the composer (the Souvenir-grade tier word floors imply this
apparatus exists), OR change the toggle to a 2-position Skim /
Reading toggle when no Deep-extra apparatus is present. The third
button is dead weight as long as Standard ≡ Deep.

Easier path: ship a small `<aside class="dig-deeper">` for at least
one per-section per Deep-tier guide so the third button has a
visible job. The compose-script Trip 2 has 8+ history paragraphs
where a "dig deeper into Norwegian sovereign-wealth-fund mechanics"
aside would fit naturally.

### Finding 3 — Top bar gets cramped at intermediate widths (~600–900px)

At the auto-narrow viewport (672×959 captured first), the top
sticky bar puts the breadcrumb crumb on the left, mode toggle in the
middle, and the floating "Save as PDF" pill on the right. At 672px
the toggle's right edge (537px) sits 14px shy of the PDF button's
left edge (551px) — they don't overlap, but visually it reads as
crowded.

The CSS hides the side TOC under 760px per spec, but the top bar
doesn't have a corresponding adjustment. A mid-breakpoint move:
collapse the breadcrumb to just the trip-name short form, OR move
the mode toggle into a small overflow menu under ~900px.

This is the only responsive issue I caught. The mobile breakpoint
itself (a single column under 600px) wasn't tested in this pass —
worth adding to a future mobile pass.

### Finding 4 — Visual hierarchy in `things_to_do` cards is inverted

When a `things_to_do` card has a venue title above a neighborhood
mono tag (e.g. "Vigeland Sculpture Park" / `Frogner`), the
neighborhood reads as more prominent because it carries the accent
color while the title reads as flat ink. Combined with Finding 1,
the title — which IS the link and should be the dominant element —
ends up the *least* visually marked element on the card.

**Phase 2b proposal:** if Finding 1 is fixed (underline becomes
visible), the hierarchy resolves itself. Worth re-checking after that
change before adding any new typographic weight.
